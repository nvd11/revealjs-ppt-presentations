What is a RAG Agent?

Evolution: From a Brain to an Entity

  • Pure LLM = 🧠 Brain

    Pain points: Knowledge blind spots, outdated info, and severe hallucinations.

  • RAG (Retrieval-Augmented Gen) = 🧠 Brain + 📚 Knowledge Base

    Pros: Provides factual evidence, turning "closed-book" into "open-book" answering.

  • RAG Agent = 🧠 Brain + 📚 Knowledge Base + 🛠️ Tools/Limbs

    Ultimate Form: Autonomously plans and executes API operations, beyond just answering.

🎯 Core Value

Radically solves LLM hallucinations and data staleness. Achieves true automation of complex business workflows within enterprise networks.

Deconstructing RAG: Core Components

Let's revisit our previous demo code. It can be distinctly split into two roles:

# Goal: Fetch the relevant private context.

Enterprise Abstraction:

In production, these two components are highly encapsulated to seamlessly interface with massive Vector DBs and various LLM engines.

Knowledge Base & Retriever: The OOP Bond

The Principle: Storage format dictates retrieval logic. We start with a contract:


from abc import ABC, abstractmethod

class BaseRetriever(ABC):
    @abstractmethod
    def retrieve(self, query: str) -> str: 
        pass # All subclasses must implement this

Form 1: Text Files ➡️ Requires Disk I/O


class FileRetriever(BaseRetriever):
    def retrieve(self, query: str) -> str:
        with open("knowledge.txt", "r", encoding="utf-8") as f:
            return extract_info(query, f.read())

Form 2: Relational DB (MySQL) ➡️ Requires SQL Execution


class DatabaseRetriever(BaseRetriever):
    def retrieve(self, query: str) -> str:
        cursor = self.db.cursor()
        cursor.execute("SELECT text FROM docs WHERE topic = %s", (query,))
        return cursor.fetchone()[0]

🤔 The Dilemma: Thanks to Polymorphism, our Generator doesn't care where the data comes from. BUT, for millions of unstructured PDFs, disk scanning is too slow, and SQL LIKE is semantically blind.
➡️ We need a revolutionary subclass: VectorDatabaseRetriever.

Vectorization & Embedding: Semantic Matching

❌ The Limits of Text Search (e.g., SQL LIKE)

  • Query: "Nutrition of an Apple"
  • Database has: "Pear" and "Rice".
  • Result: Traditional searches fail because "Apple" doesn't share characters with "Pear". They cannot comprehend that Apple and Pear are semantically closer (fruits) than Rice.

✅ Vectorization (High-Dimensional Space)

Converts natural language into arrays of floats (e.g., 768 or 1536 dimensions). In this spatial coordinate system, semantically similar words cluster together.

Embedding Models (The Encoders)

Models like OpenAI `text-embedding-3` or `BGE-M3` do this magic.


class OpenAIEmbedder(BaseEmbedder):
    def embed_text(self, text: str) -> list[float]:
        # Returns a vector like: [-0.012, 0.045, 0.111, ...]
        return openai_client.embeddings.create(input=text, model="text-embedding-3-small").data[0].embedding

The Art of Chunking & Pre-processing

Why do we need Chunking?

  • Noise & Token Limits: Passing a 100-page "Employee Handbook" PDF to answer "How many paid leave days?" causes severe LLM hallucinations (confusing sick leave/maternity leave) and blows up Token limits.
  • Traceability (Top-K): Enterprise RAG requires citations. e.g., "Answer: 5 days. [Source: Handbook P12, Paragraph 3] (Relevance: 0.95)". We must break documents into fine-grained chunks to retrieve specific paragraphs.

Strategies & Overlap

  • Methods: By Character, by Token, or by Semantic structure (e.g., Markdown headers).
  • Overlap: Prevents a long sentence from being brutally sliced in half. We keep overlapping characters (e.g., Size=500, Overlap=50) to maintain semantic continuity between adjacent chunks.

The Ingestion Pipeline (OOP Approach)


# 1. Define the Splitter
class RecursiveTextSplitter(BaseSplitter):
    def __init__(self, chunk_size=500, overlap=50):
        self.chunk_size = chunk_size
        self.overlap = overlap

# 2. Pipeline: Document -> Chunks -> Embeddings
splitter = RecursiveTextSplitter(chunk_size=500, overlap=50)
embedder = OpenAIEmbedder() # Reusing from the previous slide

chunks = splitter.split_document(full_pdf_text)

Vector DB & Schema Design

What is a Vector Database?

Optimized for storing and rapidly querying high-dimensional vector data via similarity metrics (e.g., Cosine Similarity, ANN).
Top Choices: PostgreSQL (pgvector), Google AlloyDB, BigQuery, Milvus.

Table 1: `document_chunks`

  • Fields: id, doc_id, content (TEXT), embedding (VECTOR).
  • Why store BOTH Vector and Text? The embedding is strictly for the Retriever to calculate spatial distance. However, the LLM cannot read float arrays. We must return the raw content text to the Generator for "open-book" reasoning.

Table 2: `document_topics` (Metadata Pre-filtering)

The "Recipe vs. Washing Machine" dilemma: When asking "How to cook ribs?", querying the entire DB might accidentally surface washing machine manuals. By applying a SQL WHERE topic = 'recipe' filter BEFORE the vector search, we drastically improve speed and eliminate cross-domain hallucinations. 


-- ⚠️ Crucial Step 0: The user's query must FIRST be embedded into a vector 
-- using the EXACT SAME Embedding Model as the knowledge base.
-- e.g., query_vector = embedder.embed("How to cook ribs?") -> '[0.11, 0.04, ...]'

-- Hybrid Search: Metadata Pre-filtering + Vector Similarity
SELECT dc.content, 1 - (dc.embedding <=> '[0.11, 0.04, ...]') AS similarity_score
FROM document_chunks dc
JOIN document_topics dt ON dc.doc_id = dt.doc_id
JOIN topics t ON dt.topic_id = t.topic_id
WHERE t.topic_name = 'recipe' -- 1. Filter out irrelevant domains instantly
ORDER BY dc.embedding <=> '[0.11, 0.04, ...]' -- 2. K-NN Vector Search
LIMIT 3;

The Final Piece: Generator & Top-K Tuning

The Generator's Job: Prompt Assembly

It combines the User Query with the Retrieved Chunks to force the LLM into "open-book" mode.


def generate_answer(user_query, retrieved_chunks):
    # 1. Anti-Hallucination System Prompt
    prompt = "You are a smart assistant. STRICTLY base your answer on the [References] below. Do not fabricate.\n\n"
    
    # 2. Inject Context with Traceability
    for i, chunk in enumerate(retrieved_chunks):
        prompt += f"[Reference {i+1}] (Score: {chunk.score}): {chunk.content}\n"
    
    prompt += f"\nUser Query: {user_query}"
    
    # 3. LLM Inference
    return llm_client.chat(prompt)

The Top-K Dilemma (How many chunks to feed?)

  • K is too small (e.g., K=1): Low Recall. Missing crucial context leads to incomplete answers.
  • K is too large (e.g., K=20):
    1. 💸 Expensive & Slow: Massive token consumption and latency.
    2. 🚫 Context Window Overflow: Crashing the LLM limits.
    3. 😵 Lost in the Middle: Excessive noise causes the LLM to forget the original query or ignore the context.
  • Enterprise Best Practice: Usually set K between 3 and 5.

Orchestration: The RAG Agent

Architecture Decision: LCEL vs. Imperative

Tutorials often use 1-line LCEL (chain = retriever | prompt | llm). While terse, it's a black box for debugging. For enterprise systems, we advocate Imperative Python for transparency, easy debugging, and conditional routing (like Early Exits).

Dependency Injection (OOP at its best)


class RAGAgent:
    # Decoupled! The agent doesn't care if it's PGVector or Pinecone, Gemini or GPT.
    def __init__(self, retriever: BaseRetriever, generator: ILLMGenerator):
        self.retriever = retriever
        self.generator = generator

    async def ask(self, question: str, top_k=3, topic_filters=None) -> RAGResponse:
        # 1. Retrieval Phase
        retrieved_contexts = await self.retriever.ainvoke(
            input=question, config={"top_k": top_k, "topic_filters": topic_filters}
        )
        
        # 💡 High-light: Early Exit (Prevents hallucination & saves massive API costs)
        if not retrieved_contexts:
            return RAGResponse(query=question, answer="No relevant context found.", contexts=[])

        # 2. Generation Phase (Executes ONLY if context is found)
        answer = await self.generator.ainvoke({"context": retrieved_contexts, "question": question})
        return RAGResponse(query=question, answer=answer, contexts=retrieved_contexts)

The Elegant Assembly


# 1. Instantiate the specific subclasses
my_retriever = PgVectorRetriever(db_conn)
my_generator = GeminiGenerator(model_name="gemini-2.5-pro")

# 2. Inject them into the Orchestrator
# (If we switch to GPT-4 tomorrow, the RAGAgent code doesn't change a single line!)

# 3. Execute
result = await agent.ask(question="How to cook ribs?", topic_filters=["recipe"])
