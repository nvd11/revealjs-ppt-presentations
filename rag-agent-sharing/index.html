What is a RAG Agent?

Evolution: From a Brain to an Entity

  • Pure LLM = 🧠 Brain

    Pain points: Knowledge blind spots, outdated info, and severe hallucinations.

  • RAG (Retrieval-Augmented Gen) = 🧠 Brain + 📚 Knowledge Base

    Pros: Provides factual evidence, turning "closed-book" into "open-book" answering.

  • RAG Agent = 🧠 Brain + 📚 Knowledge Base + 🛠️ Tools/Limbs

    Ultimate Form: Autonomously plans and executes API operations, beyond just answering.

🎯 Core Value

Radically solves LLM hallucinations and data staleness. Achieves true automation of complex business workflows within enterprise networks.

Why RAG? The Limits of LLMs

❌ Pain Point: Missing "Private" Knowledge


# Querying a standard LLM without context
user_query = "How many daughters does Jacky have?"
response = llm.invoke(user_query)

# 🤖 LLM Response: 
# "I cannot answer this question. There is not enough context 
# to determine which 'Jacky' you are referring to."

✅ The Breakthrough: Injecting Context (The Essence of RAG)


# 1. Provide the private knowledge as context
system_prompt = "Context: Jacky has 4 children in total, and 2 of them are boys."

# 2. Combine context with the query
final_prompt = f"{system_prompt}\nQuestion: {user_query}"
response = llm.invoke(final_prompt)

# 🤖 LLM Response: 
# "Jacky has 2 daughters."

🤔 The Real Challenge: In an enterprise with millions of documents, how does the system automatically fetch EXACTLY this "Jacky" sentence to act as context?
➡️ Let's deconstruct the architecture...

Deconstructing RAG: Core Components

Let's revisit our previous demo code. It can be distinctly split into two roles:


# 🔍 Role 1: The Retriever (Hardcoded in the demo)
# Goal: Fetch the relevant private context.
system_prompt = "Context: Jacky has 4 children in total, and 2 of them are boys."

# 🧠 Role 2: The Generator
# Goal: Combine context + query, then let LLM reason the answer.
final_prompt = f"{system_prompt}\nQuestion: {user_query}"
response = llm.invoke(final_prompt)

Enterprise Abstraction:

In production, these two components are highly encapsulated to seamlessly interface with massive Vector DBs and various LLM engines. 


# 1. Instantiate the Retriever (connects to Enterprise Vector DB)
retriever = VectorStoreRetriever(db_connection)

# 2. Instantiate the Generator (contains LLM engine & Prompt strategy)
generator = AnswerGenerator(llm_engine)

# 3. The Core RAG Pipeline
query = "How many daughters does Jacky have?"
context = retriever.retrieve(query)          # [Retrieval Phase]
answer = generator.generate(query, context)  # [Generation Phase]

Knowledge Base & Retriever: The OOP Bond

The Principle: Storage format dictates retrieval logic. We start with a contract:


from abc import ABC, abstractmethod

class BaseRetriever(ABC):
    @abstractmethod
    def retrieve(self, query: str) -> str: 
        pass # All subclasses must implement this

Form 1: Text Files ➡️ Requires Disk I/O


class FileRetriever(BaseRetriever):
    def retrieve(self, query: str) -> str:
        with open("knowledge.txt", "r", encoding="utf-8") as f:
            return extract_info(query, f.read())

Form 2: Relational DB (MySQL) ➡️ Requires SQL Execution


class DatabaseRetriever(BaseRetriever):
    def retrieve(self, query: str) -> str:
        cursor = self.db.cursor()
        cursor.execute("SELECT text FROM docs WHERE topic = %s", (query,))
        return cursor.fetchone()[0]

🤔 The Dilemma: Thanks to Polymorphism, our Generator doesn't care where the data comes from. BUT, for millions of unstructured PDFs, disk scanning is too slow, and SQL LIKE is semantically blind.
➡️ We need a revolutionary subclass: VectorDatabaseRetriever.

Vectorization & Embedding: Semantic Matching

❌ The Limits of Text Search (e.g., SQL LIKE)

  • Query: "Nutrition of an Apple"
  • Database has: "Pear" and "Rice".
  • Result: Traditional searches fail because "Apple" doesn't share characters with "Pear". They cannot comprehend that Apple and Pear are semantically closer (fruits) than Rice.

✅ Vectorization (High-Dimensional Space)

Converts natural language into arrays of floats (e.g., 768 or 1536 dimensions). In this spatial coordinate system, semantically similar words cluster together.
